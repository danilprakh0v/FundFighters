# Technical Report — FundFighters
*Курсовой проект, ФКН ВШЭ (НИС) — Прахов Данил*

## 1. Общее архитектурное решение платформы

**Client (iOS App)**
Приложение разделено на модули по экранам (Onboarding, Splash, Auth, Dashboard, Battle) по паттерну **MVVM (Model-View-ViewModel)**. 
- UI строится исключительно через код (programmatic AutoLayout), что решает проблемы мердж-конфликтов `Main.storyboard`, а также обеспечивает гранулярный пиксельный контроль над расположением элементов.
- Разработана система дизайн-токенов (`DesignSystem.swift`), концентрирующая типографику (шрифты `Golos Text`, `Inter`), отступы и корпоративные цвета проекта (например, `DS.accent` — фирменный зеленый оттенок `#1E8C62`).
- Анимации выведены на первый план (`UIKit Dynamics`, `UIView.animate` с пружинной демпинговой настройкой, сложная каскадная прозрачность во EntryAnimationViewController) для геймификации.

**Backend (ASP.NET Web API)**
Чистая архитектура (Clean Architecture / Onion) на .NET:
- **Presentation (API)**: Контроллеры для обработки HTTP-запросов и отдачи JSON.
- **Application**: Интерфейсы сервисов (таких, как отправка почты `IEmailService`), транспортные DTO, бизнес-логика.
- **Domain**: Ключевые сущности (Entities) - User, Transactions, Battles, Savings Goals.
- **Infrastructure**: Конкретная реализация БД-репозиториев (PostgreSQL + EF Core Code-First), сторонние интеграции (SMTP для отправки email с OTP или ссылками подтверждения). В частности, в `appsettings.json` хранится ключ Gmail App Password для рассылки: `smtp.gmail.com:587`.

## 2. Подробности интеграций

### Авторизация
Выполнена настройка SMTP через `MailKit`. Система поддерживает верификацию Email с генерацией случайного кода, который кешируется в памяти/БД и сверяется при вводе пользователем перед созданием аккаунта. (Включает UI-форму из 6-ти OTP квадратиков `VerificationViewController`).

### Компонентный Dashboard
Вместо монолитного экрана, Dashboard разбит на самодостаточные "уровни":
1. `BalanceCardView` — стек общей аналитики с градиентами.
2. `SavingsGoalCardView` — витрина текущего монстра-цели. Сложный расчет констрейнтов из числа 💔 ("сердец босса").
3. `RecentActivityView` & `ExpenseChartView` & `BattleCardView` — динамические карточки с фильтрацией по дате и заглушками (Mock/Placeholder), если транзакции в выбранный день отсутствуют.

### Battle View Controller (Сайд-система сражений)
Изолированная логика `Fight` (переход на выделенный экран), где через кастомные `Blur` и `Z-Index` реализовано управление интерфейсом боя (сражение с целью, получение очков/урона боссу ради сбережения определенной суммы денег). Логика сражений предусматривает визуальные эффекты урона (анимация потряхивания).

## 3. Дальнейшие задачи и перспективы
В текущей итерации создан мощный "MVP" (Minimal Viable Product) фронтенда и базовый слой бэкенда. Предстоит:
1. Интеграция слоя сетевых вызовов (URLSession Services / Mоya) с бекендом.
2. Прокидывание JWT-токена в Header запросов.
3. Замена `Mock`-данных, зашитых в ViewModel компонентов Dashboard, на реальные JSON Payload-ответы.
4. Внедрить CoreData или Realm для кэширования профиля пользователя и транзакций оффлайн.

Проект продемонстрировал возможности современного UIKit в создании производительных, эстетичных приложений, сопоставимых по качеству с SwiftUI, но без компромиссов на совместимость и гибкость AutoLayout.
